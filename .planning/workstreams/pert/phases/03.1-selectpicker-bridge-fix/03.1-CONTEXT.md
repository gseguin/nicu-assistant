<domain>
## Phase Boundary

**In scope:** Replace the bidirectional `bind:value` + dual-`$effect` bridges in `src/lib/pert/PertInputs.svelte` (medication / strength / formula) with a race-free wrapper so picker clicks persist AND external mutations to `pertState.current.*` (D-11 cascade, `pertState.reset()`, localStorage rehydration) propagate into the picker UI. After this phase, the 4 picker-driven happy-path e2e tests deferred from Phase 3 Plan 03-04 ship green, closing PERT-TEST-05 fully.

**Out of scope:**
- Any change to `src/lib/shared/components/SelectPicker.svelte` API or internals (used by feeds / gir / uac-uvc — must stay byte-identical to keep zero-regression promise on the five shipped calculators).
- Any change to `pertState` (state.svelte.ts) shape, schema, or persistence contract — Phase-1-frozen.
- Any change to the calc layer (`src/lib/pert/calculations.ts`) or the xlsx-canonical math (D-15 / D-16 from Phase 2). Pure UI-bridge fix.
- Any change to `pert-config.json` or D-11 semantics (medication change still resets strengthValue to null).
- Design polish, copy changes, OKLCH retuning, identity hue work — those belong in Phase 4.

**Anchor scope statement:** A clinician at `/pert` clicks "Creon" → `pertState.current.medicationId === 'creon'` AND the picker trigger displays "Creon" (not the placeholder), in BOTH directions (click → state, state → UI). Same for strength + formula pickers. D-11 strength-reset still fires on medication change. No other observable behavior changes.

<decisions>
## Implementation Decisions

### Resolution architecture (the fix shape)

- **D-01 [auto-recommended, KI-1 Option 2]:** Adopt the **`$derived`-backed binding wrapper** pattern for all three SelectPicker bridges in `PertInputs.svelte` (medication / strength / formula). Replace each `let xxxStr = $state<string>(...)` + read-`$effect` + write-`$effect` triplet with a single derived getter/setter pair the bind:value uses directly. Svelte 5 supports `bind:value={() => getStr(), (v) => setStr(v)}` syntax (function-binding form). This eliminates the bidirectional `$effect` race at the source — there is no longer an "effect registration order" because there are no effects driving the bridge; the getter reads `pertState.current.*` reactively, and the setter writes back synchronously inside the bind callback.
  - **Why this option (not Option 1 or 3):**
    - Option 1 (add `onValueChange` callback prop to SelectPicker) touches a shared component used by 4 calculators (feeds / gir / uac-uvc / pert) — violates the workstream's zero-regression-on-shipped-calculators promise without a careful migration plan, and there's no current evidence the other three calculators need the new API.
    - Option 3 (fork SelectPicker into `<PertSelectPicker />`) creates a divergent UI component — higher long-term maintenance, breaks the "shared-components-first" pattern from CLAUDE.md.
    - Option 2 keeps the SelectPicker contract intact (`value = $bindable()` stays as-is), localizes the change to `PertInputs.svelte` only (~50 lines deleted, ~30 lines added), and is the Svelte 5 idiomatic pattern for bridging a `string` bindable to a `string | null` / `number | null` state field.
  - **Concrete shape per bridge:**
    ```svelte
    <SelectPicker
      label="Medication"
      bind:value={
        () => pertState.current.medicationId ?? '',
        (v: string) => { pertState.current.medicationId = v === '' ? null : v; }
      }
      options={medicationOptions}
      placeholder="Select medication"
    />
    ```
  - **For strength** (number ↔ string), the setter parses with `parseInt(v, 10)` and only writes if `Number.isFinite(parsed)`; empty-string sets to `null`. Mirrors the current write-effect logic (PertInputs.svelte:108-119) but folded into the setter.
  - **For formula** (string ↔ string), trivially analogous to medication.

### SelectPicker shared-component contract

- **D-02 [auto-recommended]:** **Zero change** to `src/lib/shared/components/SelectPicker.svelte`. The `value = $bindable()` API and the `select(v)` write path stay byte-identical. This preserves the zero-regression promise for feeds / gir / uac-uvc and avoids re-running their full test/clinical-gate suites. Verifier asserts the file diff on SelectPicker.svelte is empty for this phase.
  - **Why:** Svelte 5's function-binding form `bind:value={getter, setter}` is fully compatible with any `$bindable()` prop — the consumer wraps the binding, the component sees a normal writable value. No internal change required.

### D-11 strength-reset preservation

- **D-03 [auto-recommended]:** The D-11 strength-reset effect (`PertInputs.svelte:139-147` — tracks `lastMedId`, clears `pertState.current.strengthValue = null` on medication change) **stays as-is** in its current standalone-`$effect` form. The new derived-binding wrappers don't conflict with it because the wrappers don't use `$effect` at all — they're pure derived getters + synchronous setters. When D-11 mutates `pertState.current.strengthValue = null` externally, the strength wrapper's getter (`pertState.current.strengthValue === null ? '' : String(...)`) reactively re-reads on the next microtask and the SelectPicker re-renders with the empty string → "Select strength" placeholder. No bidirectional clobber.
  - **Acceptance:** `PertInputs.test.ts:75-86` (the D-11 reset test) stays green WITHOUT any test-side change. If the test needs even a `Promise.resolve()` flush wait shift, that's a regression — the wrapper must satisfy D-11 with the same call shape the test currently uses.

### External-mutation propagation contract (the second half of the bug)

- **D-04 [auto-recommended]:** The fix MUST also handle external mutations to `pertState.current.medicationId` / `strengthValue` / `tubeFeed.formulaId` from non-picker sources:
  1. `pertState.reset()` (sets all three to `null`) — picker UI must clear to placeholder.
  2. localStorage rehydration on mount (`pertState.init()` in `+page.svelte` reads `nicu_pert_state` and assigns to `pertState.current.*`) — picker UI must reflect the persisted selection.
  3. D-11 cascade (medication change clears strengthValue) — strength picker UI must clear.
  - The derived-getter pattern (D-01) handles all three automatically because the getter reads `pertState.current.*` reactively — any external mutation triggers re-evaluation. Verifier asserts these three propagation paths via component test (TEST-3.1-04 in the plan, see D-06 below).

### Regression test strategy

- **D-05 [auto-recommended]:** Unskip the 4 picker-driven happy-path e2e tests deferred from Phase 3 Plan 03-04. They live commented-in-file at the location they would have been (`e2e/pert.spec.ts` — see KI-1 disposition). The test count goes from 8/8 → 12/12 (4 unique tests × 2 viewports = 4 + 4). NO new e2e tests authored in this phase — the deferred ones are sufficient coverage for the bug they were originally designed to catch.
  - **Tests being un-skipped:**
    1. "Oral mode happy path: weight + fat + lipase + medication + strength → hero shows capsulesPerDose" × {mobile 375, desktop 1280}
    2. "Tube-Feed mode happy path: weight + formula + volume + lipase + medication + strength → hero shows capsulesPerDay" × {mobile 375, desktop 1280}

- **D-06 [auto-recommended]:** Add **2 new component tests** (in `PertInputs.test.ts`, additive — Phase 3 Wave 1's 7 cases stay green):
  1. **Click-persist test** — Mount `<PertInputs />`, simulate click on medication SelectPicker option "Creon" via the SelectPicker's exposed select API or by dispatching the same DOM event the picker uses internally. Assert `pertState.current.medicationId === 'creon'` AND the SelectPicker trigger text reflects "Creon" (not placeholder). Mirror for strength and formula.
  2. **External-mutation propagation test** — Mount `<PertInputs />`, set `pertState.current.medicationId = 'creon'` directly (mimicking localStorage rehydration), assert SelectPicker trigger renders "Creon" within one microtask (`await Promise.resolve()`). Then call `pertState.reset()`, assert trigger renders "Select medication" placeholder.
  - **Why component tests in addition to e2e:** the e2e tests catch the symptom (visible click revert); the component tests catch the **mechanism** (so future regressions to the derived-binding pattern fail fast in vitest, not in the slower Playwright suite). Mirrors the Phase 2 / Phase 3 layering pattern.

### Backwards compatibility for non-PERT calculators

- **D-07 [auto-recommended]:** The other three SelectPicker callers (`src/lib/feeds/FeedAdvanceInputs.svelte:55-74`, plus any gir / uac-uvc usages — to be confirmed by researcher) are **NOT changed in this phase**. Their existing string-bridge pattern keeps working because (a) we don't change SelectPicker.svelte, and (b) they don't have the bidirectional-race scenario (per KI-1 root-cause: feeds' bridged field is never mutated externally during normal operation).
  - **Verifier check:** `git diff` between phase start and end shows changes ONLY in `src/lib/pert/PertInputs.svelte`, `src/lib/pert/PertInputs.test.ts`, and `e2e/pert.spec.ts` (uncommenting the 4 deferred tests). Any other touched file is a Rule-4 violation.

### Code-quality / em-dash carry-over

- **D-08 [auto-recommended]:** The 2 em-dash hits in `e2e/pert.spec.ts` DEFERRED comment block (lines 65, 70 — flagged by Phase 3 Plan 03-05 verifier, marked as documentation-style and precedent-carved-out per Phase 2) get **deleted as a side effect** of un-skipping those tests (the comment block goes away when the tests come back). ### Risk: Svelte 5 function-binding syntax availability

- **D-09 [auto-recommended, researcher to verify]:** D-01 assumes Svelte 5 `^5.55.0` (the project's pinned version per CLAUDE.md) supports the `bind:value={getter, setter}` function-binding form. This was introduced in Svelte 5.0 stable. **Researcher MUST verify** during Phase 3.1 research by:
  1. Checking the Svelte 5 docs for the current `bind:` directive contract.
  2. Locating any existing usage of function-binding in the codebase OR confirming via a minimal test compile that `bind:value={() => x, (v) => { x = v; }}` is accepted by `pnpm svelte-check` against the project's actual Svelte version.
  - **Fallback if function-binding NOT available** in `^5.55.0`: switch to a `$derived` wrapper with an explicit `$effect` writeback, but with a **single** effect (not two) — the writeback effect runs with `untrack()` around the read of `pertState.current.*` to break the bidirectional dependency. Researcher proposes the exact shape; planner locks it.
  - **Why flag this:** the entire fix architecture rests on this syntax existing. If it doesn't, the plan needs a different leaf node (still Option 2 in spirit — single race-free derived bridge — but different mechanics). Better to surface the risk now than have research/planning rediscover it.

## Existing Code Insights

### Reusable Assets

- **`$bindable()` rune** on `SelectPicker.svelte:9` — Already accepts function-binding wrappers without component change (the foundation D-01 + D-02 rest on).
- **`$derived` rune** — Used 5+ places in `SelectPicker.svelte` itself (`selectedLabel`, `hasGroups`, `groups`, `normalizedQuery`, `isSearching`, `filteredOptions`, `showGrouped`) — establishes the codebase pattern D-01 mirrors.
- **D-11 lastMedId effect** (`PertInputs.svelte:139-147`) — Standalone effect, NOT part of the bridge. Survives the bridge rewrite untouched (D-03).
- **Phase 3 Wave 1 test infrastructure** (`PertInputs.test.ts` already imports `pertState`, `render`, `tick`) — D-06 reuses verbatim; no new test setup.
- **Phase 3 Wave 3 e2e infrastructure** (`e2e/pert.spec.ts` already has the 4 dropped tests' Playwright incantations commented in-file) — D-05 un-comments; no new selectors / locators authored.

### Established Patterns

- **String-bridge proxy** (the pattern Phase 2 D-15 ported from `FeedAdvanceInputs.svelte:55-74`) — Diagnosed as the bug source by KI-1. Phase 3.1 supersedes it for PERT only; feeds keeps the legacy pattern (D-07).
- **Single-effect-with-`untrack()`** — Used elsewhere in the codebase (e.g., `+page.svelte` mount/init flow) when one direction of a reactive chain must be broken. Fallback path D-09 references this pattern.
- **`$bindable()` + getter/setter binding** (Svelte 5 idiom) — The pattern D-01 codifies for PERT.
- **Component-test-then-e2e layering** (Phase 2 → Phase 3) — D-05 + D-06 maintain it; component tests catch the mechanism, e2e tests catch the symptom.

### Integration Points

- **`PertInputs.svelte` mount sites** — `+page.svelte` mounts `<PertInputs />` twice (desktop sticky aside + mobile InputDrawer), both binding to the same `pertState` singleton (Phase 2 D-04 / 02-CONTEXT.md). The fix MUST work for both mount points; the e2e tests cover both viewports so this is regression-tested.
- **`pertState.init()` (localStorage rehydration)** — Called in `+page.svelte` on mount, mutates `pertState.current.*` directly. D-04 requires the new bridge to propagate these mutations into the picker UI.
- **`pertState.reset()`** — Wired to a "Reset" button (PertCalculator.svelte). D-04 requires the new bridge to propagate the reset into the picker UI.
- **D-11 cascade** — When `medicationId` changes (via picker click OR external mutation), the lastMedId effect clears `strengthValue = null`, which the strength bridge propagates into the picker UI (clears to "Select strength"). All three propagation paths must work.

### Architectural Notes (from KI-1 root-cause)

- The bug is **NOT** in SelectPicker. SelectPicker correctly writes `value = v` on click (line 98). The bug is in the consumer's bridge that has two `$effect`s with a registration-order dependency.
- The bug **only manifests for fields with external mutations** (PERT's `medicationId` is mutated by D-11 cascade + reset + rehydration; feeds' `enteralKcalPerOz` is only mutated by user input, never externally — that's why feeds works "by luck").
- There is **no race-free way to use two `$effect`s for a bidirectional bridge**. The fix path is to either (a) eliminate the two-effect pattern (Option 2, chosen), (b) eliminate the bidirectional binding (Option 1, rejected — touches shared component), or (c) eliminate the shared SelectPicker (Option 3, rejected — increases divergence).

## Deferred Ideas

- **Migrate feeds / gir / uac-uvc to the same derived-binding pattern.** Currently the three other calculators use the original string-bridge pattern that has the latent bidirectional race (per KI-1: it doesn't manifest because their bridged fields aren't externally mutated). - **Lint/codemod to forbid two-effect bidirectional bridges.** A custom ESLint or stylistic-rule pass that flags `let x = $state(...); $effect(() => { x = ...read... }); $effect(() => { ...write... = x; })` patterns. Would have caught KI-1 in code review. Defer to a tooling phase outside v1.15.
