## Deviations from Plan

### Auto-fixed Issues

**1. [Rule 1 - Bug] Plan-recommended selector `/Medication/i` matched 2 buttons in the rendered DOM**

- **Found during:** Task 1 verification (gate 6 — vitest run)
- **Issue:** The plan and RESEARCH §Test Strategy recommended `screen.getByRole('button', { name: /Medication/i })` to locate the medication SelectPicker trigger. In the rendered DOM (default empty state), `getByRole` threw `Found multiple elements with the role "button" and name `/Medication/i``. The matching elements:
  - Medication picker: accessible name `"Medication Select medication"` (label + placeholder)
  - Strength picker: accessible name `"Strength Choose medication first"` (label + the strength picker's "no medication selected yet" placeholder, which contains the word "medication")
  Both buttons satisfy the case-insensitive substring `/Medication/i` regex.
- **Fix:** Switched both new tests to `/^Medication/` (anchored at start of the accessible name). The Medication picker's accessible name starts with "Medication"; the Strength picker's starts with "Strength" — disambiguation is unambiguous. This mirrors the helper pattern in `src/lib/shared/components/SelectPicker.test.ts:24` (`new RegExp('^' + label)`).
- **Files modified:** `src/lib/pert/PertInputs.test.ts` only (the new tests were the source of the issue; no existing test affected).
- **Outcome:** Identical SC-1 + SC-3 behavioral assertions; the regex change is purely a selector hardening that the plan's recommended selector failed to anticipate because RESEARCH did not enumerate the Strength picker's placeholder text in its DOM-shape inventory.
- **Recommendation for future plans:** When a phase introduces multiple sibling SelectPickers whose placeholders share keywords, use anchored regexes (`/^Label/`) as the default rather than substring matches. Document the anchored form in the SelectPicker.test.ts helper as the canonical pattern (already present at line 24).

## Function-Binding Bridge Coverage Established

This plan provides the regression guards that lock in the function-binding bridge behavior shipped by Plan 03.1-01:

- **D-01 covers the user-input direction** (the original visible bug — picker click reverted to placeholder). The test exercises `<SelectPicker>`'s `select(v)` write path on line 99 of `SelectPicker.svelte` flowing through the function-binding setter at PertInputs.svelte line 192-193 (`(v: string) => { pertState.current.medicationId = v === '' ? null : v; }`) and back into the trigger's `selectedLabel` `$derived` re-render.
- **D-04 covers the external-mutation direction** (the second half of the bug; broken by the second hotfix attempt during Phase 3). Together, the 2 new component tests (run sub-second in vitest jsdom) catch the bridge mechanism. The Wave 2 sibling Plan 03.1-03 will catch the user-visible symptom via Playwright e2e.
