## Architectural Responsibility Map

| Capability | Primary Tier | Secondary Tier | Rationale |
|------------|-------------|----------------|-----------|
| Picker click → state mutation | Browser (Svelte runtime in client) | — | Pure client component reactivity. SelectPicker writes `value = v` synchronously inside an `onclick`; the bridge's setter writes through to `pertState.current.*`. |
| State mutation → picker UI | Browser (Svelte runtime + `$derived` re-eval) | — | Getter is a pure function `() => pertState.current.X ?? ''` — Svelte's reactivity tracks the read; any mutation re-evaluates on next microtask flush. |
| State persistence | Browser (localStorage) | — | `pertState.persist()` runs from `$effect`-watched `JSON.stringify`. Unaffected by this phase. |
| State rehydration | Browser (localStorage on `pertState.init()`) | — | `pertState.init()` runs in `PertState` constructor (eager). After rehydration, the new bridges' getters re-read the rehydrated values on first render. |

All capabilities live in the **browser tier**. No server, API, CDN, or DB involvement. This phase has zero cross-tier risk.

## Approach (Recommended Fix Shape)

Per D-01: replace lines 74-134 of `PertInputs.svelte` (the three string-bridge proxy blocks: 60 lines) with three inline `<SelectPicker>` invocations using function bindings (~36 lines including comments). Total net delta: **−24 lines** in `PertInputs.svelte`.

**Architecture in plain words:**
- The wrapper getter `() => pertState.current.medicationId ?? ''` reads `pertState` reactively. When `medicationId` mutates from ANY source (picker click via the setter, external mutation via D-11 cascade or `reset()` or `init()`), Svelte's runtime detects the read dependency and re-runs the getter on the next microtask flush. SelectPicker re-renders with the fresh value.
- The wrapper setter `(v) => { pertState.current.medicationId = v === '' ? null : v; }` runs synchronously inside the SelectPicker's `select(v)` callback (which writes `value = v` on line 99 of SelectPicker.svelte). The setter writes directly to `pertState`, eliminating the intermediate `$state<string>` proxy that caused the original race.
- **No `$effect` is involved in the bridge.** The two-effect race is gone because there are no effects driving the bridge. `pertState.persist()` and the D-11 `lastMedId` effect are unrelated standalone effects that survive untouched.

**Why this beats the swap-effect-order attempts that failed during Phase 3:**
- Hotfix attempt 1 (swap order, write-first) broke D-11 because the swapped strength write-effect raced against the external `strengthValue = null` mutation.
- Hotfix attempt 2 (fold D-11) broke external-write propagation on medication.
- The new pattern has **no `$effect` at all** for the bridge → no registration order → no race. The setter runs in a callback (not an effect), so external mutations have no competing scheduled work to clobber them.

### Canonical syntax to lock into the plan

```svelte
<!-- Single-quoted parenthesized 2-arg form. The PARENS around the inner pair
     are required by Svelte's parser to distinguish from a single arg. -->
<SelectPicker
  label="Medication"
  bind:value={
    () => pertState.current.medicationId ?? '',
    (v: string) => { pertState.current.medicationId = v === '' ? null : v; }
  }
  options={medicationOptions}
  placeholder="Select medication"
/>
```

## Common Pitfalls

### Pitfall 1: Forgetting parens around the function-binding pair
**What goes wrong:** Writing `bind:value={() => x, (v) => x = v}` without outer parens around inner pair, or merging into one arrow, gives a syntax / semantics error.
**Why it happens:** The 2-arg form is uncommon syntax; muscle memory wants single arrow.
**How to avoid:** Use the canonical Svelte 5 docs form exactly: `bind:value={\n  () => getter,\n  (v: type) => { setter; }\n}`. The comma between getter and setter is the key delimiter; Svelte's parser uses it.
**Warning signs:** `pnpm svelte-check` complains about `bind:value` expecting an lvalue, or a parse error on the comma.

### Pitfall 2: Setter performs comparison-then-write (not needed)
**What goes wrong:** Adding `if (next !== pertState.current.X) pertState.current.X = next;` inside the setter — vestigial pattern from the deleted bridges.
**Why it happens:** Carrying forward the equality guards that the old bidirectional bridge needed to prevent ping-pong.
**How to avoid:** The function-bind setter runs ONLY on user-driven `select(v)` calls from SelectPicker. There is no ping-pong loop to guard against. Direct assignment (`pertState.current.X = v === '' ? null : v;`) is sufficient AND idiomatic.
**Warning signs:** Code looks heavier than the Approach §Code Examples templates above.

### Pitfall 3: Strength setter swallowing legitimate empty-string deselect
**What goes wrong:** A setter that only writes on `Number.isFinite(parsed)` skips the empty-string case → strength stays stale when D-11 cascade fires `strengthValue = null` then user re-selects the (now-different) medication.
**Why it happens:** Copying the parseInt-then-isFinite block from the old write-effect verbatim without the empty-string early return.
**How to avoid:** Use the explicit `if (v === '') { ... return; }` guard at the top of the setter, as shown in the Strength template above.
**Warning signs:** The new external-mutation-propagate test (D-06 case 2) for the strength picker fails; or the D-11 reset test (existing line 75-86) regresses.

### Pitfall 4: Asymmetric test for D-06 click-persist (only checks state, not UI)
**What goes wrong:** The test asserts `pertState.current.medicationId === 'creon'` but doesn't assert the trigger text changes. The OLD bug was specifically that state went to `null` AND the placeholder reverted — both halves matter.
**Why it happens:** Easier-to-write assertion; sufficient-feeling.
**How to avoid:** Both assertions are mandatory: `expect(pertState.current.medicationId).toBe('creon')` AND `expect(trigger.textContent).toContain('Creon')`. Mirror SelectPicker.test.ts T-10 + T-07 verbatim.
**Warning signs:** Test passes but visual regression slips through.

### Pitfall 5: jsdom rune-batching variance (already mitigated in Phase 3)
**What goes wrong:** Single `await tick()` doesn't flush rune dependencies in some jsdom + node version combos.
**Why it happens:** Phase 3 D-11 test (line 83-84) discovered this; the fix is double-await-Promise-resolve as defensive guard.
**How to avoid:** Use `await tick(); await Promise.resolve();` after every state mutation in the new D-06 tests, mirroring the precedent.
**Warning signs:** Test is flaky locally vs CI; only fails in `pnpm test:run` non-watch mode.

### Pitfall 6: E2e dialog click locator ambiguity at mobile viewport
**What goes wrong:** PertInputs is mounted twice (`+page.svelte` desktop sidebar + mobile drawer). Both bind to the same singleton, so `pertState` updates work, but locator queries can match BOTH `<button>Medication</button>` triggers, giving Playwright a strict-mode violation.
**Why it happens:** The `getInputsScope` helper handles input-element scoping but not necessarily the modal `<dialog>` overlay that opens GLOBALLY (not scoped to the aside / drawer container).
**How to avoid:** Use `scope.getByRole('button', { name: /^Medication/ })` to click the correct trigger (scoped). Then use `page.getByRole('option', { name: /^Creon$/ })` (page-level, NOT scoped) because the picker's `<dialog>` is rendered as a top-level modal — only one is open at a time, so page-level option lookup is unambiguous.
**Warning signs:** Playwright "strict mode violation: locator resolved to N elements" error.
