## Deviations from Plan

### Auto-fixed Issues

**1. [Rule 1 - Bug] Plan template used `getByRole('button', ...)` for the searchable Formula picker trigger; actual rendered role is `combobox`**
- **Found during:** First pert.spec run after the initial commit; both Tube-Feed test cases timed out at line 102 waiting for `aside[aria-label="PERT inputs"]>>getByRole('button', { name: /^Formula/ })`.
- **Issue:** The plan's literal template (per RESEARCH §E2E Mechanics) used `scope.getByRole('button', { name: /^Formula/ })` for the Formula picker trigger. RESEARCH's locator inventory was correct for the Medication + Strength pickers (which are non-searchable, role=undefined → resolved to `button`) but missed that SelectPicker.svelte:172 sets `role={searchable ? 'combobox' : undefined}` — the Formula picker is `searchable=true`, so its trigger role is `combobox`, NOT `button`. The Playwright accessibility snapshot from the failing test confirmed: `combobox "Formula Select formula" [ref=e70]`.
- **Fix:** Switched the Formula trigger locator to `scope.getByRole('combobox', { name: /^Formula/ })`. Added a 3-line inline comment citing SelectPicker.svelte:172 so the asymmetry is documented at the test-code site.
- **Files modified:** e2e/pert.spec.ts:101-105 (single-line locator change + 3-line explanatory comment).
- **Recommendation for future plans:** RESEARCH §E2E Mechanics + plan template inventory should distinguish `searchable=true` (role=combobox) from `searchable=false` (role=button) for any SelectPicker trigger locator. The plan's CRITICAL writing rules section did not flag this.

**2. [Rule 1 - Bug] Plan template used `getByRole('searchbox')` for the Formula search input; actual rendered role is `textbox`**
- **Found during:** Second pert.spec run after fix #1; both Tube-Feed cases progressed past the Formula trigger click but timed out at the next step waiting for `getByRole('searchbox')`.
- **Issue:** The plan's literal template (per RESEARCH §E2E Mechanics) used `page.getByRole('searchbox')` for the search input. The plan's deviation protocol explicitly anticipated this: "If the role inference fails (Playwright reports no matching searchbox), fall back to `page.getByRole('textbox', { name: /Filter Formula/i })` (the input has `aria-label='Filter Formula'` per SelectPicker.svelte:204)." The Playwright accessibility snapshot from the failing test confirmed: `textbox "Filter Formula" [active] [ref=e77]` — Playwright's `getByRole('searchbox')` resolves only `<input type="search">`; SelectPicker.svelte:201 uses `<input type="text">` so the accessible role defaults to `textbox`.
- **Fix:** Switched the search-input locator to `page.getByRole('textbox', { name: /Filter Formula/i })`. Added a 3-line inline comment citing SelectPicker.svelte:199-208 so the type='text' constraint is documented at the test-code site.
- **Files modified:** e2e/pert.spec.ts:106-110 (single-line locator change + 3-line explanatory comment).
- **Recommendation for future plans:** Either (a) the SelectPicker search input could change to `type="search"` in a future shared-component refactor (out of Phase 3.1 scope per D-02 zero-change contract), or (b) `getByRole('textbox', { name: /Filter ${label}/i })` should be the default in future test plans that touch a searchable SelectPicker.

**3. [Rule 1 - Bug] Plan template used `page.getByText('150')` (no exact-match) which strict-mode-collided with the `1500` Volume input value**
- **Found during:** Third pert.spec run after fixes #1 and #2; the Tube-Feed test reached its final assertion but failed with `strict mode violation: getByText('150') resolved to 2 elements: 1) <span>1500</span> (Volume per day) 2) <span>150</span> (Capsules per month)`.
- **Issue:** The plan's literal template asserted `await expect(page.getByText('150')).toBeVisible();` for the capsulesPerMonth value. The Volume per day input is filled with `1500`, which is rendered as `<span class="num text-ui font-medium ...">1500</span>` somewhere on the page (likely in the input box's aria-label or value display); `getByText('150')` matches both as a substring per Playwright's default contains-match semantics.
- **Fix:** Added `{ exact: true }` to the assertion: `await expect(page.getByText('150', { exact: true })).toBeVisible();`. Added a 2-line inline comment explaining the exact-match rationale.
- **Files modified:** e2e/pert.spec.ts:128-130 (single-arg change + 2-line comment).
- **Recommendation for future plans:** Any `page.getByText(numericString)` assertion in a test that has a numeric input whose value is a superstring of the expected output should default to `{ exact: true }`. The plan template did not anticipate this case.

**4. [Rule 1 - Style] 2 em-dashes introduced in my own bug-fix comments (transient self-introduced violation)**
- **Found during:** Final post-fix em-dash audit; `grep -c '—' e2e/pert.spec.ts` returned 2 (vs the expected 0 per D-08).
- **Issue:** The 3-line explanatory comments I added during fixes #1 and #2 included em-dashes (`'button' — only` and `'textbox' — getByRole`).
- **Fix:** Replaced both em-dashes with periods (`'button'. Only` and `'textbox'. getByRole`). Comment-only change; pert.spec re-run confirmed 12/12 still passing.
- **Files modified:** e2e/pert.spec.ts:103, e2e/pert.spec.ts:107.

## Decisions Made

- **D-05 reframing applied verbatim per RESEARCH:** Authored 2 new tests rather than uncommenting any (the deferred tests did not exist as commented-in-file code; CONTEXT.md D-05 was inaccurate on this point and the planner caught it).
- **D-08 closure as side effect:** The 2 em-dash hits in the deleted prose block disappeared automatically — no separate em-dash sweep task. (Note: I introduced 2 transient em-dashes in my own bug-fix comments that I caught and removed before commit; final em-dash count is 0.)
- **Locator strategy per RESEARCH Pitfall 6:** Scope-relative for triggers (`scope.getByRole('combobox' | 'button', { name: /^Label/ })`); page-level for dialog options (`page.getByRole('option', { name: /^Value$/ })`). Both forms appear in the file as required by the plan's verify gates. Anchored regex `/^Label/` prevents the Strength placeholder ("Choose medication first") from matching `/medication/i` (the asymmetry that broke Plan 03.1-02's first pass).
- **D-11 cascade wait via polled assertion (not magic timeout):** Each strength-picker click is preceded by `await expect(scope.getByRole('button', { name: /^Strength/ })).toContainText(/Select strength/);`. Playwright auto-retries up to its default 5s assertion timeout, deterministic across CI variance.

## Lessons for Future Plans

1. **Plan templates should distinguish `searchable=true` (role=combobox) from `searchable=false` (role=button) for SelectPicker triggers.** Generic "use `getByRole('button', ...)`" guidance is insufficient when one of the three picker instances has `searchable=true`. RESEARCH §E2E Mechanics inventory needs a per-instance accessibility-role column.
2. **Plan templates should default to `getByRole('textbox', { name: /Filter Label/i })` for the SelectPicker search input, not `getByRole('searchbox')`.** Until SelectPicker.svelte is updated to use `<input type="search">` (out of Phase 3.1 scope per D-02), the searchbox role is wrong.
3. **Plan templates should default to `{ exact: true }` for any `getByText(numericString)` assertion if a numeric input on the same page might hold a value that is a superstring of the expected output.** This is a clear pattern that the planner can statically check.
4. **Em-dash audits should be self-applied to comments authored DURING execution, not just to the pre-commit file state.** Bug-fix comments authored mid-run can introduce new em-dashes that cancel the side-effect-of-prose-deletion gain.
