## Deviations from Plan

### Deviation: init() unconditionally sets `_acknowledged` instead of conditionally setting only when keys exist

- **Plan T2:** suggested `if (v2 === 'true' || v1 === 'true') { _acknowledged = true; ... }` — leaves `_acknowledged` untouched when neither key is set.
- **Actual:** `_acknowledged = v2 === 'true' || v1 === 'true';` — unconditional assignment.
- **Reason:** the conditional form fails singleton-state isolation across vitest scenarios. The disclaimer module is dynamically imported per test, but the module-level `let _acknowledged = $state(false)` retains its previous value (`true` after a prior `acknowledge()` call) when the module is re-imported in a new test. The unconditional assignment ensures a fresh `init()` with empty localStorage always resets `_acknowledged` to `false`. The migration write (`if (v1 === 'true' && v2 !== 'true')`) stays conditional and idempotent — no extra writes when v2 is already `'true'`.
- **Verified:** all 9 singleton tests + 6 banner component tests pass.

### Deviation: AboutSheet placed AFTER `</div>` instead of before in +layout.svelte

- **Plan T6:** "ADD before the closing `</div>`: `<AboutSheet calculatorId={activeCalculatorId} bind:open={aboutOpen} />`"
- **Actual:** placed AFTER the `</div>` closes the layout wrapper (sibling of the flex column).
- **Reason:** AboutSheet is a bits-ui Dialog rendered to a portal — its DOM placement in source has no effect on its rendered position in the DOM (Dialog.Portal moves it to `<body>`). Placing it as a sibling of the layout wrapper (rather than inside it) is cleaner code organization — the layout wrapper holds visible flow content; portal-managed dialogs sit outside.
- **Verified:** AboutSheet still opens correctly via banner "More" link (e2e test 4) AND via hamburger menu's About row (covered by existing hamburger tests, none of which broke).

### Deviation: removed unused `activeCalculatorId` $derived + unused `CalculatorId` type import from NavShell

- **Plan T6:** specified removing the AboutSheet import + mount and the local `aboutOpen` state. Did not explicitly mention removing the now-unused `activeCalculatorId` derivation.
- **Actual:** removed both. After AboutSheet is hoisted, NavShell has no remaining consumer for `activeCalculatorId`. Leaving dead code would add noise without benefit. The same logic was added to +layout.svelte (where the live consumer now lives).
- **Verified:** `pnpm check` passes 0/0 — no orphaned references.

### Deviation: e2e migration test added explicit hydration wait

- **Plan T7:** described the v1 -> v2 migration scenario without specifying timing.
- **Actual:** the migration test waits for `localStorage.getItem(KEY_V2) === 'true'` via `page.waitForFunction` after navigation, before asserting v2 is written. Without this wait, the assertion races the onMount-driven `disclaimer.init()` call.
- **Reason:** in the fresh-install scenarios the banner-visible assertion implicitly waits for hydration (visibility check). In the migration scenario the banner is NEVER visible (because v1 was true), so `toHaveCount(0)` resolves before hydration finishes. The explicit `waitForFunction` gates the next assertion on the migration write completing.
- **Verified:** test 3 passes consistently (3 runs).

### Deviation: axe sweep scoped to banner subtree

- **Plan T8:** "Add an axe assertion in the banner spec (or extend existing axe sweeps) for the banner-visible state in both themes per UI-SPEC §axe Sweep Scope item 2."
- **Actual:** added the axe assertion via `AxeBuilder.include('[aria-label="Clinical use disclaimer"]')` so the sweep runs ONLY against the banner subtree.
- **Reason:** an unscoped axe sweep on `/morphine-wean` with banner visible would re-trigger pre-existing baseline failures unrelated to this phase (e.g., bits-ui Slider on the page would flag if the route had one — and the global a11y sweeps documented above already track those failures). Scoping to the banner subtree gives a clean, focused signal — "is the banner accessible?" — without coupling this test to unrelated baselines.
- **Verified:** both light and dark axe sweeps pass with 0 violations against the banner subtree.
