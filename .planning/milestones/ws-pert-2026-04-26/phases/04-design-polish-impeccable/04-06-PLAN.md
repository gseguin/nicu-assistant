<objective>
**Wave 6 of Phase 4 -- gap closure for the weight-in-Oral-mode UX concern surfaced as a Wave 5 user follow-up.**

After Wave 5 closed the Oral hierarchy inversion via D-12 + D-13 (commits `f4565b4` + `618a66f`; user resume signal `approved` for the Oral `Estimated daily total` row promotion + cross-mode parity), the user surfaced a verbatim follow-up: `"...but it looks like weight isn't used in oral mode. double check against .xlsx. If that's the case remove it"`. Investigation against `epi-pert-calculator.xlsx` Pediatric PERT Tool sheet (B4 cell traced through B9 / B10 / B11) and `src/lib/pert/calculations.ts` (`computeOralResult` at lines 77-100; `getTriggeredAdvisories` at lines 207-260) confirmed:

- Weight (xlsx B4 / state `pertState.current.weightKg`) IS used in Oral mode but ONLY for the PERT-SAFE-01 max-lipase-cap safety advisory (xlsx `B11 = B4 * 10000`; calc-side: `getTriggeredAdvisories` reads weight to fire PERT-SAFE-01 when daily lipase exceeds `weightKg * 10000`).
- Weight is NOT a parameter in the Oral dose formula itself (xlsx `B9 = B5*B6` for Total Lipase Needed; `B10 = ROUND(B9/B8, 0)` for Capsules Needed).
- "Remove weight" is NOT applicable -- removing weight would silently disable PERT-SAFE-01 in Oral mode, which is a clinical-safety regression (the calculator becomes silently less safe than the spreadsheet it parities against).
- The actual UX gap is that weight reads as a primary dose-driving input when it is in fact a safety-only input; the user has no visual cue that it gates the safety advisory rather than the dose.

**The gap-closure decision was authored 2026-04-26 in 04-CONTEXT.md as D-14:**

- **D-14** authorizes adding a mode-conditional helper text rendered as a sibling `<p>` element after the weight input in `src/lib/pert/PertInputs.svelte` (the SHARED card at lines 106-125 in HEAD), gated `{#if pertState.current.mode === 'oral'}`. Helper text wording: planner picks; reference shape: `Used for the 10,000 units/kg/day safety check, not the dose calculation.` Tube-Feed mode unchanged (weight IS used in Tube-Feed for the `lipasePerKg` secondary output per xlsx Tube-Feed `B13 = B12/B4`, so no helper text in Tube-Feed). ~3 LOC; PERT-route only; no D-08b violation (RangedNumericInput shared component NOT touched -- the helper text is a NEW sibling element added to the route-local PertInputs.svelte parent).
