<objective>
Wave 1 of Phase 4 -- Design Polish (`/impeccable` Critique Sweep).

This plan runs the `/impeccable critique` skill against the live `/pert` UI in 8 contexts (theme × viewport × mode = 2×2×2 per CONTEXT D-01) to produce 8 transcript artifacts + 1 normalized triage table + the auto-disposition recommendation that drives Wave 2.

**Three discrete pieces of work:**

1. **Pre-flight setup** (PRODUCT.md authoring + AUDIT.sh skeleton): the `/impeccable` skill MUST have PRODUCT.md at the repo root per SKILL.md §Setup or the critique degrades to generic output (RESEARCH Pitfall 1). PRODUCT.md is authored manually from CLAUDE.md §"Design Context" per RESEARCH Option A (the `--auto` orchestrator cannot run the interactive `/impeccable teach` fallback). The DESIGN.md named-rule audit script (UI-SPEC D-05a) is also dropped at the phase directory now so Wave 3 can run it without re-authoring it.

2. **Critique sweep** (the work): start dev server, run 8 `/impeccable critique http://localhost:5173/pert` invocations one per (theme, viewport, mode) triple, capture each transcript to `04-CRITIQUE-{theme}-{viewport}-{mode}.md`, then run the deterministic CLI scan (`npx --yes -p impeccable@latest impeccable detect --json` against the 3 PERT route files) → 04-DETECT.json. Tear down dev server.

3. **Triage** (the deliverable): normalize all 8 transcripts + CLI output into `04-FINDINGS.md` triage table per UI-SPEC §"Findings normalization" columns. Auto-assign dispositions per CONTEXT D-03 rules (P1 → fix-now no exceptions; cheap P2 → fix-now; expensive P2 / shared-component-touching P2 → defer; P3 → defer by default). Compute aggregate score + Wave-2 plan-count recommendation. Write 04-01-SUMMARY.md.

**Why this is one plan with three tasks** (not three separate plans): Task 1 (PRODUCT.md + AUDIT.sh) is one-shot setup; Task 2 (critique sweep) requires Task 1's PRODUCT.md to be in place AND must be sequential within itself (single dev-server window state); Task 3 (FINDINGS normalization) reads ALL 8 transcripts produced by Task 2. None of the three pieces can run in parallel; combining them in one plan honors the sequential dependency without manufacturing fake parallelism.

**Why Wave 2 is a separate placeholder plan** (CONTEXT D-07a replan beat): the Wave 2 plan count cannot be authored ahead of Wave 1 output. Wave 2 fix scope = N plans where N depends on the P1 + cheap-P2 finding distribution. The orchestrator (or `/gsd-plan-phase 4 --gaps`) reads 04-01-SUMMARY.md's Wave-2 recommendation to author the actual Wave 2 plans post-Wave-1.

Output:
- 1 NEW repo-root file: PRODUCT.md (boundary acknowledgment in SUMMARY)
- 1 NEW phase-dir bash script: 04-AUDIT.sh (executable, 10-rule audit)
- 8 NEW phase-dir markdown transcripts: 04-CRITIQUE-{theme}-{viewport}-{mode}.md
- 1 NEW phase-dir JSON: 04-DETECT.json (deterministic CLI seed)
- 1 NEW phase-dir markdown: 04-FINDINGS.md (triage table + Wave-2 recommendation)
- 1 NEW phase-dir markdown: 04-01-SUMMARY.md (audit trail)
- ZERO production code or test code modifications (Wave 1 is read-only against the live UI)

Per CONTEXT D-08 / D-08a, the only D-08a amendment introduced by this plan is `PRODUCT.md` at repo root (acknowledged in SUMMARY). All other artifacts live under `.planning/`, which is implicitly outside the negative-space allowlist scope (`src/` + `e2e/`).
