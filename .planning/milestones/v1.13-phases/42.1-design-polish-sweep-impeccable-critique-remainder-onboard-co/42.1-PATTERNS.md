## How to Use

When implementing each file, the planner agent and the executor agent should follow the closest existing analog cited here. This PATTERNS.md does **NOT** replace `DESIGN.md` — `DESIGN.md` is the visual contract; this is the implementation pattern map. When the analog and the visual contract conflict, the visual contract wins; record the deviation in the plan summary.

Two ground rules borrowed from `42-PATTERNS.md`:
1. **Mirror the closest analog rather than invent a new pattern.** The five calculator surfaces have already converged enough that the right move is consolidation, not novelty.
2. **Where no shipped analog exists for a NEW component, use the closest reasonable starting point and call it out explicitly.** Both NEW components in this phase fall in that category — see the per-file sections below.

## Special Patterns to Honor

### The HeroResult adoption pattern

When refactoring a calculator to use `<HeroResult>`, the executor must follow this exact sequence:
1. **Import** `HeroResult` from `'$lib/shared/components/HeroResult.svelte'` at the top of the calculator's `<script>`.
2. **Move pre-formatting out of the markup** into `$derived` values: `let displayValue = $derived(formatMg(rawValue))` so the HeroResult call site shows clean prop wiring.
3. **Replace the existing hero markup** (a `<section class="card animate-result-pulse" style="background: var(--color-identity-hero);" aria-live="polite" aria-atomic="true">…</section>` block) with a single `<HeroResult eyebrow={...} value={...} unit={...} pulseKey={...} />`. Use the `children` Snippet override path for non-default shapes (morphine 3-cell summary, GIR two-line, feeds bedside multi-row).
4. **REMOVE the existing `aria-live` wiring** at the calculator wrapper level. HeroResult itself owns `aria-live="polite" aria-atomic="true"`. Duplicate live regions in the same subtree degrade screen-reader announcements.
5. **REMOVE the existing `{#key}` block** if present. HeroResult internally `{#key pulseKey}`s the children; wrapping outside is redundant.

### The DisclaimerBanner migration pattern

The migration is one-time and idempotent (per RESEARCH finding 1; CORRECTS CONTEXT D-14's bare-key reference to actual `_v1` key shipping in `disclaimer.svelte.ts:3`):

```typescript
const DISCLAIMER_KEY_V1 = 'nicu_assistant_disclaimer_v1';
const DISCLAIMER_KEY_V2 = 'nicu_assistant_disclaimer_v2';

init(): void {
  const v2 = localStorage.getItem(DISCLAIMER_KEY_V2);
  const v1 = localStorage.getItem(DISCLAIMER_KEY_V1);
  if (v2 === 'true' || v1 === 'true') {
    _acknowledged = true;
    if (v1 === 'true' && v2 !== 'true') {
      try {
        localStorage.setItem(DISCLAIMER_KEY_V2, 'true');
        // Do not delete v1 — preserves audit trail
      } catch { /* private mode */ }
    }
  }
  _initialized = true;
}

acknowledge(): void {
  _acknowledged = true;
  try { localStorage.setItem(DISCLAIMER_KEY_V2, 'true'); } catch {}
}
```

### The Identity-Inside conformance test

A new vitest test enforces D-01's doctrine via static-source regex. Pattern (no exact analog; novel):

```ts
import { describe, it, expect } from 'vitest';
import { readFileSync } from 'fs';
import { resolve } from 'path';

describe('Identity-Inside Rule conformance (D-01)', () => {
  it('NavShell.svelte chrome paths do not reference --color-identity', () => {
    const src = readFileSync(resolve('src/lib/shell/NavShell.svelte'), 'utf8');
    // Permissive: identity may appear inside CSS variable definitions, not in chrome class strings
    const offenders = src.match(/border-\[var\(--color-identity\)\]|text-\[var\(--color-identity\)\]/g);
    expect(offenders, `NavShell chrome must use --color-accent (Clinical Blue), not --color-identity`).toBeNull();

  });
});
```

This test makes the doctrine self-enforcing — adding a new chrome surface that uses identity color will fail CI.
